# 校园问答 LLM 红蓝队攻防演示方案

> 适用场景：课堂/组会展示 MITM Prompt 注入与网络层防御。
> 本方案基于当前仓库最新代码（MITM 注入开关 + 网络层蓝队代理 + 应用层防御）。

---

## 一、演示目标

让观众直观理解：

1. **红队**：中间人可以在用户无感知的情况下篡改 LLM 请求，注入恶意指令。
2. **蓝队**：在网络层检测到注入载荷后，可以清洗或阻断请求，保护后端模型。
3. **应用层防御**：即使注入到达模型，输出守卫也能拦截异常响应。

---

## 二、课前准备（本机测试版）

### 2.1 环境要求

- Windows 电脑一台
- Conda 环境 `kimi_chat`（Python 3.11）
- 已安装 `mitmproxy`
- 已配置 `.env` 中的 `LLM_API_KEY`
- 已安装 mitmproxy CA 证书（系统受信任的根证书颁发机构）

### 2.2 一键启动代理链

```powershell
cd A:\The_code\kimi_chat
conda activate kimi_chat
git pull origin main
.\start_redblue_chain.ps1 -AttackMode prompt
```

成功后会看到：

```text
Blue web UI: http://127.0.0.1:8085
Red  web UI: http://127.0.0.1:8083 (password: mitm)
```

### 2.3 启动受害者 Streamlit

再开一个 PowerShell 窗口：

```powershell
cd A:\The_code\kimi_chat
conda activate kimi_chat
$env:HTTP_PROXY="http://127.0.0.1:8082"
$env:HTTPS_PROXY="http://127.0.0.1:8082"
streamlit run app.py
```

## 三、角色分工

| 角色 | 负责内容 | 对应界面 |
|---|---|---|
| 受害者 | 在 Streamlit 正常提问 | http://localhost:8501 |
| 红队（攻击者） | 运行 MITM 代理，注入 prompt | http://127.0.0.1:8083 |
| 蓝队（防御者） | 运行网络层防御代理，清洗/阻断 | http://127.0.0.1:8085 |

> 本机测试时三个角色都在同一台电脑上；双机演示时红/蓝队在攻击者电脑，受害者在另一台。

## 四、演示流程（建议 10~15 分钟）

### 步骤 0：打开三个面板

- Streamlit：`http://localhost:8501`
- 红队 mitmweb：`http://127.0.0.1:8083`（密码 `mitm`）
- 蓝队 mitmweb：`http://127.0.0.1:8085`

在两个 mitmweb 顶部 Filter 框输入：

```text
~u api.xiaomimimo.com
```

---

### 步骤 1：正常场景（红队关、蓝队关）

**操作**：

- 侧边栏关闭红队和蓝队开关。
- 输入问候语：`你好`。
- 输入校园问题：`东南大学图书馆几点开门？`

**预期效果**：

- 模型正常打招呼、正常回答校园问题。
- 证明系统在没有攻击时工作正常。

---

### 步骤 2：红队 MITM 注入（红队开、蓝队关）

**操作**：

- 打开 **红队攻击模式**。
- 攻击分类：**直接注入**。
- 攻击方法：**system_override**。
- 关闭 **蓝队防御模式**。
- 输入校园问题：`东南大学图书馆几点开门？`

**预期效果**：

- Streamlit 中模型回答异常，可能输出系统提示词或承认“不再受限制”。
- 在红队面板中点击请求，查看 **Request → Text**，能看到用户问题后面被追加了 `[SYSTEM OVERRIDE]` 注入载荷。

> 如果模型对齐较强、拒绝执行，可以告诉观众：现代模型有一定抵抗力，但攻击载荷仍在流量中，这就是 MITM 威胁所在。

---

### 步骤 3：蓝队应用层防御（红队开、蓝队开、网络层动作=阻断）

**操作**：

- 打开 **蓝队防御模式**。
- 蓝队网络层动作：**阻断请求（block）**。
- 再次输入：`东南大学图书馆几点开门？`

**预期效果**：

- Streamlit 助手返回：

  ```text
  🛡️ 【网络层防御拦截】蓝队代理检测到提示词注入攻击，已阻断该请求。
  命中规则：注入标记: [SYSTEM OVERRIDE]; 忽略之前的指令
  ```

- 请求没有到达 LLM API，蓝队代理直接返回了防御响应。
- 在蓝队面板中能看到请求被拦截，Response 内容就是上述防御消息。

---

### 步骤 4：蓝队网络层清洗（红队开、蓝队开、网络层动作=清洗）

**操作**：

- 把蓝队网络层动作改成 **清洗载荷（sanitize）**。
- 再次输入：`东南大学图书馆几点开门？`

**预期效果**：

- Streamlit 中模型正常回答校园问题，就像没有攻击一样。
- 在蓝队面板中点击请求，查看 **Request → Text**，注入载荷已被剥离，只剩下正常问题。
- 请求头中多了：

  ```text
  X-Blue-Team-Sanitized: true
  ```

---

### 步骤 5：编码混淆攻击与防御（可选）

**操作**：

- 红队攻击分类改成 **编码混淆**。
- 攻击方法：**base64_encode**。
- 蓝队网络层动作保持 **阻断** 或 **清洗**。
- 输入：`东南大学图书馆几点开门？`

**预期效果**：

- 红队面板中 Request 里的注入是 Base64 编码的，肉眼不易识别。
- 蓝队 addon 会解码 Base64 并检测到隐藏指令，进行阻断/清洗。
- 展示编码混淆不是“免死金牌”，网络层仍然可以检测。

---

## 五、演示收尾

1. **关闭红队开关**，再发一次校园问题，确认恢复正常。
2. **总结**：
   - MITM 可以在用户无感知时篡改请求。
   - 网络层防御可以在流量层面清洗/阻断注入。
   - 应用层防御是最后一道防线。
3. **安全提示**：课堂演示后记得更换 API Key，因为 Bearer Token 在代理中以明文可见。

## 六、常见问题

| 问题 | 解决 |
|---|---|
| 红队/蓝队面板打不开 | 访问 `http://127.0.0.1:8083` 或 `http://127.0.0.1:8085`，不要用 `0.0.0.0`。 |
| 面板里没有流量 | 确认 Streamlit 窗口设置了 `HTTP_PROXY` 指向 `127.0.0.1:8082`。 |
| 模型拒绝注入 | 说明模型对齐较强，可以切换到更直接的 `system_override` 再试，或强调“载荷已在流量中”。 |
| 端口被占用 | 用 `Get-Process mitmweb \| Stop-Process` 结束旧代理。 |
| Streamlit 说“无法回答” | 确认已 `git pull` 最新代码，并重启 Streamlit。 |

## 七、双机演示（可选）

如果想在课堂上用两台电脑演示：

- **攻击者电脑**：开热点，运行 `start_redblue_chain.ps1`。
- **受害者电脑**：连接热点，安装 mitmproxy CA，设置系统代理为 `192.168.137.1:8082`，运行 `streamlit run app.py`。
- 攻击者面板地址对受害者不可见，主要在大屏或攻击者电脑上展示。
