# MCP 协议安全攻防 — 攻击者 vs 受害者 双机对抗流程

## 角色说明

| 角色 | 电脑 | 人数 | 任务 |
|------|------|------|------|
| **攻击者 (Attacker)** | 电脑A | 1人 | 开热点、运行 mitmproxy + Wireshark、拦截篡改流量 |
| **受害者 (Victim)** | 电脑B | 1人 | 连接热点、运行 Streamlit、正常与 AI 交互 |
| **主持人/讲解者** | 任意 | 1人 | 控制节奏、讲解原理、展示大屏 |

> ⚠️ **法律与道德提示**：本实验仅用于授权的教学场景，所有参与者必须知情同意。不要在未授权的真实网络中实施 ARP 欺骗或中间人攻击。

---

## 前期准备

### 硬件

- 2 台 Windows 电脑
- 1 个投影仪（可选，用于向全班展示）

### 软件

| 软件 | 安装在 | 用途 |
|------|--------|------|
| Python 3.11 | 攻击者 + 受害者 | 运行 Streamlit / mitmproxy |
| mitmproxy | 攻击者 | 中间人代理、拦截篡改 |
| Wireshark | 攻击者 | 网络层抓包展示 |
| conda | 受害者 | 管理 Python 环境 |
| Git | 受害者 | 拉取项目代码 |

### 网络

- 攻击者电脑需要能上网（推荐插网线）
- 攻击者电脑能开 WiFi 热点

### Round 2：配置代理并启动 Streamlit（3分钟）

#### 受害者操作

```powershell
# 1. 进入项目目录
cd campus-qa-redblue

```

#### 攻击者操作

- 确认 mitmproxy Web 界面已打开：`http://localhost:8081`
- Wireshark 已启动

---

### Round 3：正常请求演示（2分钟）

#### 受害者操作

1. 在浏览器中打开 `http://localhost:8501`
2. 切换到 **🔌 MCP 协议攻防**
3. 选择 **🌐 真实 API Server**
4. 工具选择 `get_weather`
5. 输入城市：`Nanjing`
6. 点击 **▶️ 调用工具**

#### 预期结果

显示南京真实天气：
```json
{
  "data": {
    "city": "Nanjing",
    "temperature_c": "22",
    "weather_desc": "Sunny"
  }
}
```

#### 攻击者讲解

> "这是正常调用，请求经过 mitmproxy，但我们先不拦截。Wireshark 中已经能看到这个 HTTP 包了。"

---

### Round 4：请求篡改攻击（3分钟）

#### 攻击者操作

1. 在 mitmproxy Web 界面中找到 `wttr.in/Nanjing` 请求
2. 点击该请求右侧的 **Intercept** 按钮
3. 请求被暂停，显示详细信息
4. 在 URL 路径中把 `Nanjing` 改成 `Beijing`
5. 点击 **Resume** 放行

#### 受害者界面变化

明明输入的是 `Nanjing`，但结果显示：
```json
{
  "data": {
    "city": "Beijing",
    "temperature_c": "18",
    "weather_desc": "Cloudy"
  }
}
```

#### 主持人讲解

> "看到没有？受害者查的是南京，但攻击者把请求改成了北京。这就是 MCP 中间人攻击中的请求篡改！"

---

### Round 5：响应篡改攻击（3分钟）

#### 攻击者操作

1. 让受害者再次输入 `Nanjing` 并调用
2. 这次不拦截请求，让它正常发到 wttr.in
3. 在 Response 返回时点击 **Intercept**
4. 找到 JSON 中的 `temp_C` 字段
5. 把真实温度改成 `45`
6. 点击 **Resume** 放行

#### 受害者界面变化

显示：
```json
{
  "data": {
    "city": "Nanjing",
    "temperature_c": "45",
    "weather_desc": "Sunny"
  }
}
```

#### 主持人讲解

> "这次请求没改，但响应被改了！南京 45°C，明显是假的。如果这是医疗诊断或金融数据，后果会非常严重。"

---

### Round 6：Wireshark 抓包展示（2分钟）

#### 攻击者操作

1. 在 Wireshark 中设置过滤：
   ```
   host wttr.in
   ```
2. 找到被篡改的请求包
3. 右键 → 追踪流 → TCP 流
4. 展示完整的 HTTP 请求和响应

#### 展示重点

```
GET /Beijing?format=j1 HTTP/1.1
Host: wttr.in
X-Exfiltrated-By: malicious-mcp-proxy
...
```

> "大家看 Wireshark 里，这个 GET 请求明明是 /Beijing，但受害者输入的是南京。网络层确实被篡改了。"

---

### Round 7：数据外泄展示（2分钟）

#### 攻击者操作

1. 在 mitmproxy 中查找 POST 请求（如果有 exfiltrate 模式）
2. 或者展示 Query 参数中的敏感信息
3. 展示攻击者可以看到的完整数据：
   - 用户查询的城市
   - 查询时间
   - 请求头中的 User-Agent
   - 甚至 IP 地址

#### 主持人讲解

> "中间人不仅能改，还能看。你的每次查询、每个参数，攻击者都能记录下来。这就是数据隐私风险。"

## 第三阶段：防御演示（3分钟）

### 关闭攻击

#### 攻击者操作

```powershell
# 1. 按 Enter 关闭 start_attacker.ps1
# 2. 关闭 Wireshark
# 3. 关闭 Windows 热点
```

#### 受害者操作

```powershell
# 1. 关闭 Streamlit（Ctrl+C）
# 2. 关闭系统代理
#    设置 → 网络和 Internet → 代理 → 关闭
# 3. 删除 mitmproxy 证书
#    运行 certmgr.msc → 受信任的根证书 → 删除 mitmproxy 证书
```

### 主持人讲解防御方案

1. **HTTPS 证书固定（Certificate Pinning）**
   - App 内置信任的证书，不随意信任 mitmproxy 的 CA

2. **请求签名**
   - 每个请求带数字签名，篡改后签名验证失败

3. **代理检测**
   - App 检测是否运行在有代理的环境中，弹出警告

4. **端到端加密**
   - 请求体和响应体加密，中间人无法读取明文

5. **MCP 防御模块**
   - 像我们项目里的 `defense/mcp_defense.py` 一样，检测危险权限和工具

## 故障排查

### 问题1：受害者连不上热点

- 检查热点密码是否正确
- 关闭防火墙试一下
- 重启热点

### 问题2：mitmproxy 看不到流量

- 检查受害者是否设置了正确的代理 IP 和端口
- 检查 mitmproxy 是否绑定到 0.0.0.0
- 检查防火墙是否允许 8080 端口

### 问题3：HTTPS 请求报错

- 受害者必须安装 mitmproxy 证书
- 确认安装到了"受信任的根证书颁发机构"

### 问题4：Streamlit 页面打不开

- 检查 `streamlit run app.py` 是否启动成功
- 浏览器访问 `http://localhost:8501`
- 检查 conda 环境是否激活

### 问题5：Wireshark 看不到 wttr.in 流量

- 确认 Wireshark 选择了热点网卡
- 过滤条件是否正确：`host wttr.in`
- 确认请求真的发出了（不是被缓存）

## 总结

| 角色 | 核心操作 | 关键技能 |
|------|---------|---------|
| 攻击者 | 开热点 → 启动 mitmproxy → 拦截篡改 → Wireshark 展示 | 网络代理、流量分析 |
| 受害者 | 连热点 → 设代理 → 启动 Streamlit → 正常交互 | 配置代理、使用 Streamlit |
| 主持人 | 讲解原理、控制节奏、引导互动 | 表达能力、安全意识 |

这个流程设计让所有观众都能直观理解 MCP 中间人攻击的危害和防御思路。
